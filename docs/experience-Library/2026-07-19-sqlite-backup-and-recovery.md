# SQLite 版本迁移、自动备份与可恢复替换

## 现象

学习状态等新功能需要修改旧项目数据库。如果直接执行 `ALTER TABLE`，一旦迁移中断、磁盘异常或后续代码有误，用户已经积累的人工修改和学习进度可能无法恢复。直接复制正在使用的 SQLite 文件也可能得到不一致快照。

## 原因

- SQLite 数据库可能正在写入，文件级复制无法保证事务一致性。
- 旧版本没有学习状态字段，必须在首次打开时向前迁移。
- 恢复操作本身也是高风险写入，不能直接覆盖唯一的正式文件。

## 修复

1. 迁移前通过 SQLite `backup()` 接口建立一致性快照。
2. 每天首次打开有效项目时创建自动备份，重新导入和恢复前增加保护性备份。
3. 每个备份执行 `PRAGMA quick_check`，无效文件不出现在恢复列表。
4. 恢复时先把备份写入同目录临时数据库，检查通过后使用原子替换更新 `project.sqlite3`。
5. 默认保留最近 12 份，兼顾恢复范围与磁盘占用。

## 防复发规则

- 后续新增数据库字段时，必须先备份再迁移，并为旧结构建立自动测试。
- 不得使用普通文件复制备份正在使用的 SQLite 数据库。
- 恢复前必须保存当前有效数据库；恢复目标和临时文件必须限制在项目工作区。
- 新版本验收同时检查数据版本、旧字段保留、备份可读和恢复后的学习状态。
