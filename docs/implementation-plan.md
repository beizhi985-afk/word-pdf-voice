# v0.1 实施方案

## 技术结构

- 桌面界面：Qt/PySide6 Essentials，可随应用打包并避免依赖系统 Tcl/Tk。
- PDF 提取：`pdfplumber`，按页面表格读取四列数据。
- 本地数据：SQLite，保存原始字段、人工修改、异常标记和音频状态。
- 本地发音：`kokoro-onnx`，模型延迟加载并按词条缓存 WAV。
- Anki 导出：`genanki`，把实际音频文件打包进 `.apkg`。
- 播放：Windows 原生 `winsound`，不依赖外部播放器。

## 模块边界

1. `extractor` 只负责只读解析和结构校验。
2. `storage` 负责持久化，重新导入不得覆盖人工修改。
3. `tts` 负责可替换的语音引擎、文件命名和缓存。
4. `anki_export` 负责稳定卡片标识和媒体打包。
5. `app` 负责交互、后台任务和进度展示，不承载解析规则。

## 实施阶段

1. 先通过真实 PDF 回归测试锁定 4450 条结构。
2. 生成 30 个覆盖普通词、缩写、同形异音和疑似异常的试听样本。
3. 完成搜索、筛选、编辑、逐词播放和批量生成界面。
4. 完成 Anki 卡组导出与随机抽查。
5. 最后整理 Windows 安装和打包入口。

## 输出与隐私

- 源 PDF 不进入 Git。
- 默认工作数据保存在用户本地应用数据目录。
- 模型、音频、数据库和生成卡组均被 Git 忽略。
- 第一版本地发音不上传词汇或 PDF 内容。
